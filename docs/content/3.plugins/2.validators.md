# Validation

Nuxt Upload Kit validates files through three mechanisms, each with a distinct purpose:

1. **`restrictions`** in `nuxt.config.ts` — declarative rules (size, count, MIME type) enforced identically on the client and the server.
2. **Server-side validators** in `server/upload.server.config.ts` — request-aware checks that need server context (DB quotas, magic-byte sniffing, external API calls).
3. **Client-side processing plugins** — checks that need to run in the browser before a file is added (filename rules, async client-only validation).

## Restrictions

Restrictions are the recommended way to enforce file rules. They are configured once in `nuxt.config.ts` and applied on both the client (before adding files) and the server (when handling the upload request) — so a malicious client can't bypass them.

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  modules: ["nuxt-upload-kit"],

  uploadKit: {
    restrictions: {
      maxFiles: 10,
      maxFileSize: 50 * 1024 * 1024, // 50MB per file
      maxTotalSize: 500 * 1024 * 1024, // 500MB combined
      minFileSize: 1024, // 1KB
      allowedMimeTypes: ["image/*", "application/pdf"],
      disallowedMimeTypes: ["image/svg+xml"],
    },
  },
})
```

### Fields

| Field                 | Type       | Description                                                  |
| --------------------- | ---------- | ------------------------------------------------------------ |
| `maxFileSize`         | `number`   | Maximum size of a single file (bytes)                        |
| `minFileSize`         | `number`   | Minimum size of a single file (bytes)                        |
| `maxFiles`            | `number`   | Maximum number of files allowed                              |
| `maxTotalSize`        | `number`   | Maximum combined size of all files (bytes)                   |
| `allowedMimeTypes`    | `string[]` | Allowed MIME types. Wildcards like `image/*` and `*/*` work  |
| `disallowedMimeTypes` | `string[]` | MIME types to reject. Wildcards work                         |

All fields are optional.

### Security note: MIME enforcement is UX-grade, not security

`allowedMimeTypes` and `disallowedMimeTypes` check the client-reported `Content-Type` / file `type`. A hostile client can rename `evil.exe` to `evil.jpg` and report `Content-Type: image/jpeg` — the rule will let it through.

If the type matters for security (e.g., you serve uploaded files back to other users), pair it with content-byte sniffing in a server validator:

```ts
// server/upload.server.config.ts
import { fileTypeFromBuffer } from "file-type"

export default defineUploadServerConfig({
  storage: S3Storage({...}),
  mode: "server", // sniffing requires bytes — only available in server mode
  validators: [
    async (file, ctx) => {
      const event = ctx.event
      // ...read part bytes from event, check magic bytes...
      const detected = await fileTypeFromBuffer(bytes)
      if (!detected || file.mimeType !== detected.mime) {
        throw createError({ statusCode: 415, message: "Declared MIME doesn't match content." })
      }
    },
  ],
})
```

In `presigned` mode the server never sees the bytes, so use the **`afterUpload`** hook to fetch and check post-upload, or rely on cloud-side scanning (S3 antivirus, etc.).

### Wildcards

`allowedMimeTypes` and `disallowedMimeTypes` accept wildcard patterns:

```ts
allowedMimeTypes: [
  "image/*", // All images (jpeg, png, gif, webp, etc.)
  "video/*", // All videos
  "audio/*", // All audio
  "application/pdf", // Specific type
  "*/*", // Anything (use sparingly)
]
```

### Per-instance overrides

Pass `restrictions` to `useUploadKit` when a specific component needs different limits than the app-wide defaults. The override merges field-by-field — fields you don't specify fall back to `nuxt.config.ts`.

```ts
const avatarUploader = useUploadKit({
  restrictions: {
    maxFiles: 1,
    maxFileSize: 500_000,
    allowedMimeTypes: ["image/jpeg", "image/png"],
  },
})
```

::prose-callout{type="warning"}
Per-instance overrides are enforced on the client only. The server continues to apply the global `restrictions` from `nuxt.config.ts`, so client-side overrides should always be **stricter** than the global ones, never looser.
::

## Handling Validation Errors

When restrictions reject a file, it gets `status: 'error'`. Listen via the `file:error` event:

```ts
uploader.on("file:error", ({ file, error }) => {
  if (error.message.includes("MIME type")) {
    toast.error("Please upload images or PDFs only")
  } else if (error.message.includes("Maximum number")) {
    toast.error("Too many files. Please remove some first.")
  } else if (error.message.includes("File size")) {
    toast.error("File is too large.")
  }
})
```

Or render the error inline on the file:

```vue
<template>
  <div v-for="file in uploader.files" :key="file.id">
    <span v-if="file.status === 'error'" class="error">
      {{ file.error?.message }}
    </span>
  </div>
</template>
```

## Server-side custom validators

Use `validators: [...]` on `defineUploadServerConfig` for **server-only** checks that need request context — DB quotas, tenant isolation, magic-byte sniffing, external API calls. They run after `authorize` and after the declarative `restrictions` from `nuxt.config.ts` have passed.

```ts
// server/upload.server.config.ts
import { createError } from "h3"

export default defineUploadServerConfig({
  storage: S3Storage({...}),
  authorize: async (event) => ({ userId: event.context.user.id }),
  validators: [
    async (file, ctx) => {
      const count = await db.files.count({ where: { userId: ctx.auth.userId } })
      if (count >= 100) throw createError({ statusCode: 409, message: "Account file limit reached." })
    },
  ],
})
```

Throw `createError({ statusCode, message })` to reject — the status code is returned to the client.

## Client-side custom validators

For rules that need to run **in the browser** before a file is added — duplicate detection, filename rules, async client-side checks — write a processing plugin with a `validate` hook. Throw to reject the file.

```ts
import { defineProcessingPlugin } from "nuxt-upload-kit"

const ValidatorNoEmoji = defineProcessingPlugin(() => ({
  id: "validator-no-emoji",
  hooks: {
    validate: async (file, context) => {
      const emojiRegex = /[\u{1F600}-\u{1F64F}]/u
      if (emojiRegex.test(file.name)) {
        throw { message: "Filenames cannot contain emoji" }
      }
      return file
    },
  },
}))

const uploader = useUploadKit({
  plugins: [ValidatorNoEmoji()],
})
```

### Example: duplicate detection

```ts
import { defineProcessingPlugin } from "nuxt-upload-kit"

const ValidatorNoDuplicates = defineProcessingPlugin(() => ({
  id: "validator-no-duplicates",
  hooks: {
    validate: async (file, context) => {
      const isDuplicate = context.files.value.some((f) => f.id !== file.id && f.name === file.name && f.size === file.size)
      if (isDuplicate) {
        throw { message: "File already exists" }
      }
      return file
    },
  },
}))
```
